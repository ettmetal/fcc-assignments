Introduction

ARIA roles are a mechanism for declaring that an HTML element is performing a different role in a page or webapp than the one defined by the HTML standards. They are one part of the ARIA specs for creating accessible webpages and webapps. The specs also include a set of attributes which help convey the state of various objects defined by these roles.

This is often necessary because the role of the element does not yet have an HTML element explicitly associated with it, or may not yet have widespread support across browsers.

ARIA roles allow assigning a semantic role to an element in a page.

In order to make use of ARIA roles, you should have an understanding of HTML, including semantic markup, and attributes.

Using ARIA roles

Most ARIA roles are simple to apply to HTML code. Some roles have additional information they need to work with to be a useful representation of the element they are describing, but applying the role is as simple as adding it's name to an element's role attribute: <div role="note">.

There are some rules that guide the usage of ARIA roles, though.

  1. If you can achieve the same semantic effect with a built-in element you should do so
  2. Do not change native semantics, unless you must
  3. All ARIA elements which have interactions must support those interactions with a keyboard
  4. Do not use ARIA techniques (e.g. role="presentation") to hide focusable elements
  5. All interactive elements must have an accessible name.

Document Structure roles

Document structure roles describe the role an element plays in organising the content of a page. Most of these roles describe structure in the same way that a semantic HTML element does. Roles which don't have wide support as an HTML equivalent are:

  • toolbar
  • tooltip
  • feed
  • math
  • presentation / none
  • note

Because of the rule of using ARIA roles, it's important only to use roles which don't have an equivalent in HTML semantic tags, or when for other reasons it is not possible to use that markup.

Widget roles

These roles describe interactive patterns. Similar to Document structure roles, widget roles are often supported with standard semantic HTML elements. The difference is that interactivity of these widgets will usually need to be provided by JavaScript, regardless of if they are used from HTML or defined with ARIA roles. Many of these roles can be used by standalone elements, or as part of composites however some are reserved for describing composite widgets - often containers for other widgets.

Some of the widget roles which are not well-covered by HTML semantics are:

  • scrollbar
  • searchbox
  • separator
  • slider
  • spinbutton
  • switch
  • tab
  • tabpanel
  • treeitem
<div>
    <div role="tablist">
        <button role="tab" onclick="tabSelect(1);">Tab 1</button>
        <button role="tab" onclick="tabSelect(2);">Tab 2</button>
    </div>

    <div role="tabpanel" id="tab-1">
        <!--Content for first tab-->
    </div>

    <div role="tabpanel" id="tab-2">
        <!--Content for second tab-->
    </div>
</div>

Landmark roles

Landmark roles provide landmarks for high-level organisation of a page. Many of them will automatically apply to semantic HTML elements, such as role="navigation", which is automatically applied to <nav> elements. Some of these roles, such as role="banner" are only applied if certain conditions are met.

Typically, these are used by assistive technologies to provide quick (e.g. keyboard-driven) navigation to different parts of a page. Because of this, overuse of these roles can create a busy conceptual layout which is harder to understand.

Live region roles

Live region roles define parts of a page which can be updated dynamically and which may not be the user's primary focus. When defined this way, assistive technologies know they should "watch" these areas for changes, and inform the user of changes that happen. For example, a screen reader might announce a change in a region with one of these roles. There are five live region roles:

  • alert
  • log
  • marquee
  • status
  • timer

The most common live region role is the alert role. It is important not to overuse alerts because they are intrusive. 

<div role="alert">
  Something went wrong!
</div>

Window roles

These roles define elements which act as sub-windows which are placed within the same window. There are currently two types:

  • dialog
  • alertdialog

The alertdialog role is a combination of the alert live region role and the dialog role. The two are differentiated by their need for user interaction: if no user interaction is required, it is an alert. Otherwise, it should be an alertdialog. 

<div role="alertdialog">
  Something went wrong! We can try to continue, or cancel the operation. Which whould you like?
  <button onclick="continueOperation();">Continue</button>
  <button onclick="cancelOperation();">Cancel</button>
</div>

Abstract roles

Abstract roles are a type of role reserved for classifying the types of roles used in ARIA specifications. They are used by web browsers to aid in organising a page, but they must not be used in HTML content. These are the abstract ARIA roles:

  • command
  • composite
  • input
  • landmark
  • range
  • roletype
  • section
  • sectionhead
  • select
  • structure
  • widget
  • window
